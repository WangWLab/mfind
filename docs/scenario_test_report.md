## 场景 1: 首次打开初始化

### 测试内容
- 生成 1000 个测试文件
- 构建初始索引
- 验证服务就绪时间

### 测试结果
```
生成文件：1000 个，耗时 69ms
索引构建：1001 文件，耗时 3.89ms
速度：257,203 文件/秒
```

### 评估
✅ **通过** - 索引构建速度优秀，1000 文件仅需约 4ms

### 潜在问题
1. **每次启动都重新构建索引** - 当前 CLI 实现没有持久化存储，每次启动都会重新扫描文件系统
2. **大规模文件索引时间** - 10 万文件需要约 18ms，对于首次启动可能偏长

### 建议改进
1. 实现索引持久化，将索引保存到磁盘（如 `~/.cache/mfind/index.db`）
2. 启动时先加载缓存索引，再后台增量更新
3. 添加进度指示器

### 测试内容
- 构建索引并导出
- 多次从持久化导入索引
- 验证加速比

### 测试结果
```
导出索引大小：52,829 bytes
首次构建：1.28ms
导入恢复：51µs
加速比：25.2x
```

### 评估
✅ **通过** - 持久化导入比重新构建快 25 倍

### 潜在问题
1. **CLI 未使用持久化** - 虽然 `IndexEngine::export/import` 已实现，但 CLI 没有调用
2. **无自动保存机制** - 索引变更后不会自动保存到磁盘

### 建议改进
1. CLI 启动时检查缓存索引是否存在且有效
2. 实现索引自动保存（如每次变更后或定期保存）
3. 使用 LMDB 作为持久化存储后端

✅ **通过** - 正确检测到文件变化

1. 实现基于 inode 和 mtime 的快速变化检测
2. 增量构建索引，只处理变更的文件

## 场景 4: 后台持续运行

### 测试内容
- 启动 FSEvents 监控
- 模拟文件变化
- 验证事件捕获

### 评估
⚠️ **需改进** - 当前实现存在以下问题：

### 问题识别
1. **轮询而非 FSEvents** - 当前 `FSEventsWatcher` 使用轮询方式，不是真正的 macOS FSEvents API
2. **仅监控目录本身** - 轮询只监控传入的路径，不监控子目录
3. **无递归监控** - 无法检测子目录的文件变化

### 代码分析
```rust
// 当前实现：轮询方式
while running.load(Ordering::SeqCst) {
    thread::sleep(poll_interval);  // 轮询间隔
    for path in &watched_paths {
        // 只监控顶层路径
    }
}
```

### 建议改进
1. 实现真正的 FSEvents API（使用 `core-foundation` bindings）
2. 支持递归监控子目录
3. 优化事件批处理和去重

## 架构问题总结

### 1. 索引持久化未集成到 CLI

**问题:** `IndexEngine::export/import` 已实现，但 CLI 每次启动都重新构建索引

**影响:**
- 频繁启动场景性能差
- 用户体验不佳（每次都需等待索引构建）

**建议方案:**
```rust
// CLI 启动流程
async fn run_search() -> Result<()> {
    let cache_path = get_index_cache_path();

    let mut engine = IndexEngine::new(config)?;

    // 尝试加载缓存索引
    if cache_path.exists() {
        let data = fs::read(cache_path)?;
        engine.import(&data).await?;
        eprintln!("从缓存加载索引");

        // 后台验证和增量更新
        tokio::spawn(async move {
            verify_and_update_index();
        });
    } else {
        // 首次构建
        engine.build(&roots).await?;
        // 保存缓存
        let data = engine.export().await?;
        fs::write(cache_path, data)?;
    }

    // 执行搜索...
}
```

### 2. FSEvents 使用轮询而非原生 API

**问题:** 当前实现是轮询，效率低且不准确

**影响:**
- 事件延迟高
- 可能遗漏快速连续的文件变化
- 浪费 CPU 资源

**建议方案:** 使用 `fsevent` crate 或直接调用 Core Foundation FSEvents API

### 3. 无索引健康检查机制

**问题:** 长时间未启动后，无法判断索引是否过期

**建议方案:**
- 保存索引时的时间戳和文件 checksum
- 启动时验证索引健康状态
- 根据健康状况决定是否重新索引
